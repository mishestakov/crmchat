# Здоровье аккаунта: счётчик отправок, страйки, ручной отдых

Источник: сессия с Юлей (`specs/bd-test-tasks.md` стр.50-89) + продуктовая
проработка 24.06.26. Объединяет три «хотелки», которые на деле — одна гроздь
поверх общего фундамента (журнала событий):
- кулдаун/блок аккаунта (стр.50);
- счётчик отправок новым + адаптивный лимит (стр.81-89);
- «менеджер должен видеть, когда поймали бан за рассылки».

## Продуктовая рамка: три разных состояния, не одно

1. **Кулдаун (авто, временный)** — TG ответил FloodWait/PEER_FLOOD. Аккаунт сам
   вернётся в строй. Человек только видит «когда отпустит». Уже работает
   (`cooldownUntil`/`cooldownReason`, воркер пропускает и сам сбрасывает).
2. **Отдых/пауза (ручной, временный)** — менеджер сам решает «дать полежать N
   дней». Профилактика, не реакция. Делаем — поверх того же `cooldownUntil`.
3. **Блок (терминальный)** — `banned`/`unauthorized`. Сам не вернётся.
   `banned` уже выкидывается из пула проекта; `unauthorized` → переавторизация.

Два типа «бана» по-разному опасны:
- **FloodWait** — не бан, «притормози», TG даёт срок. Безопасно.
- **PEER_FLOOD** — антиспам на письма НОВЫМ. Повторные → реальный бан. Это и
  есть «страйк», который менеджер должен видеть. `newLeadsDailyLimit` (деф. 15,
  эмпирика Юли: 10 безопасно, 15 ок) пытается его предотвратить.

## Слой 0 — фундамент: журнал событий (СДЕЛАНО)

Таблица `outreach_account_events` (см. `apps/api/src/db/schema.ts`): одна строка
на отправку/страйк/смену состояния. Кормит и счётчик, и будущий график.
- `type`: `cold_send | peer_flood | flood_wait | banned | unauthorized |
  manual_rest | resume`.
- НЕ хранит адресата: «кому писали» живёт в leads/scheduled_messages; журналу
  нужны только числа по дням. `detail` — опц. причина/заметка.
- Чтения: счётчик = индексный `COUNT WHERE account+type+at`, график =
  `GROUP BY date`. Никакого warm-резолва на чтении.
- Запись (helper `recordAccountEvent`, `lib/account-events.ts`) в точках смены
  состояния: `cold_send` (worker, факт холодной отправки), `peer_flood`
  (worker + listener), `flood_wait` (worker + quick-send), `banned`/
  `unauthorized` (смена status), `manual_rest`/`resume` (Слой 1.5).

Нюанс семантики: живой гейт лимита в воркере считает «осталось слотов» по
ТЕКУЩЕМУ warm-статусу, а счётчик в UI читает ЗАПИСАННЫЕ события. Внутри дня
могут разойтись на 1-2, если холодный адресат успел ответить. Для показа ок,
гейт не трогаем.

## Слой 1 — видимость

В **списке** аккаунтов (строка) — только счётчик: **«Сегодня новым: X / Y · за
30 дней: Z»** (`X` = COUNT cold_send сегодня в tz, `Y` = `newLeadsDailyLimit`,
`Z` = COUNT cold_send за 30 дней). Без светофора. Историю страйков в списке НЕ
показываем — лишний шум; в списке достаточно счётчика и бейджа cooldown.

Историю — на **странице аккаунта**, сворачиваемой карточкой «История · 30 дней»
(ручка `/activity`, GROUP BY по дню в tz на 1 аккаунт — дёшево). В заголовке —
счётчик страйков (виден и свёрнутым). Внутри по дням: сколько холодных отправок
+ какие страйки/паузы (`peer_flood`/`banned`/`unauthorized` красным,
`manual_rest`/`resume`/`flood_wait` серым). Это «график по дням» в виде таблицы —
дешёвая основа под будущий настоящий график.

## Слой 1.5 — ручной пульт (страница аккаунта)

Объединено в одну карточку «Отправка» с дневным лимитом (логически вяжется):
- **Крутилка `newLeadsDailyLimit`** — поднять жирному, опустить новому.
- **Пауза** одним контролом: поле «дней» + кнопка. Модель — одно состояние:
  в строю / на паузе до DD.MM. «Продлить» = ввести новое число и применить (бэк
  берёт `max(текущий, выбранный)`). «Вернуть в строй» появляется только когда
  есть что возвращать. Пауза по таймеру с авто-возвратом (`cooldownUntil` +
  событие `manual_rest`); возврат — `clearAccountCooldown` + `resume`. Менеджер
  главнее антиспама (консистентно с тем, что ручную отправку не гейтим).

## Отложено (явно НЕ в этом срезе)

- **График** активности/страйков на таймлайне — Слой 2, нужен накопленный
  журнал.
- **Авто-адаптивный порог** (машина сама двигает лимит по корреляции
  «сколько слал → когда страйк») — Слой 3, нужны данные. Пока цифру двигает
  человек.
- 30-дневной детальной сводки нет — только «сегодня» + «за 30 дней» числом +
  список последних страйков.
- Правка пула аккаунтов в запущенном проекте — `banned` и так выкидывается,
  «отдыхающий» воркер пропускает; отдельной ручки не строим.
- Warning в `LaunchModal` (пул × здоровье) — опциональный хвост.
- Подсказку «пишите из обычного Telegram» в чат НЕ добавляем: гейт ручной
  отправки снят, менеджер пишет тёплым прямо из сервиса — подсказка
  противоречила бы этому.
