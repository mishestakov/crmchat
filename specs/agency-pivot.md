# Pivot: от Web3-аутрича к рекламному агентству

Документ фиксирует решение остановить копирование donor'а (CRMChat) и переключиться
на доменную модель рекламного агентства, которое размещает рекламу клиентов в
Telegram-каналах. См. также `DECISIONS.md` (общие архитектурные отклонения) и
`user-stories.md` (легаси US-1…US-30 по Web3-сценарию — постепенно депрекейтятся).

## Почему пивот

Web3-сценарий донора: CRM «вокруг персоны», плоская модель `contact + properties`,
один канбан = состояние сделки, кампании = транзитные источники трафика.

Агентство работает иначе:
- **Работает с инвентарём** (каналы), а не с персоной. У одного админа может
  быть N каналов; история — по обоим уровням.
- **Главная единица — проект**, а не контакт. Каждый проект клиента живёт
  2-6 недель, имеет бюджет, KPI, медиаплан, и собственный канбан размещений.
- **Один канал может одновременно жить в N проектах** на разных стадиях.
- **Существенная часть базы — повторные блогеры**. Холодные рассылки —
  периферийная активность, не центр продукта.
- **Финансовый слой first-class**: маржа, акты, два направления оплат
  (клиент → агентство, агентство → блогер).

Объём изменений: доменная модель меняется на 60-70%, UI основных экранов на 80%.

## Целевая доменная модель

```
Воркспейс (агентство)
├── Клиент (рекл): юр.лицо, договор с агентством, маржа, контактные лица
│   └── Проект (рекламная кампания клиента): бриф, бюджет, KPI, медиаплан
│       └── Размещение: канал × проект × дата
│           ├── Стадии: Подбор → Оффер → Прайс → ОК клиента → Договор →
│           │           Драфт → Правки → ЕРИД → Опубликовано → Метрики → Акт → Закрыт
│           ├── Артефакты: договор, драфт, креатив, ЕРИД, пост-URL, скрины, акт
│           └── Транзакции: блогеру (расход), от клиента (доход)
│
└── База инвентаря (cross-проектная)
    ├── Блогер: TG-аккаунт, форма работы, реквизиты, агрегаты по истории
    └── Канал: тематика/гео/размер, прайс-лист, метрики (ER, охваты)
        └── привязан к Блогеру N:1
```

### Ключевые сущности

**Клиент** — внутри воркспейса. Юр.лицо рекла + договор + дефолтная маржа.

**Проект** — принадлежит клиенту. Бриф (даты, бюджет, KPI типа CPV/CPC, ТЗ-док,
TOV, ограничения), статус (`Бриф → Подбор → Согласование подборки →
Согласование креативов → Размещения → Закрыт`), бюджет/факт/маржа.

**Блогер** — общая база агентства, не привязан к клиенту/проекту. Поля: TG-аккаунт,
имя, форма работы (физик/самозанятый/ИП/ООО), реквизиты, валюта.
Агрегаты: число проектов, открученный бюджет, средний CPV, % дедлайнов в срок,
% правок «с первого раза», стоп-флаги (скручивал, удалял раньше).
ЭДО / договор: загруженные сканы, статус подписания.

**Канал** — общая база, привязан к Блогеру N:1. Поля: ссылка, тематика,
платформа (TG/YouTube/VK), размер, гео/возраст/пол аудитории, прайс-лист
(пост/24ч/нативка/интеграция), свежие метрики (ER, охваты последних 10 постов,
динамика подписчиков), дата последнего обновления цен.

**Размещение** — пересечение `канал × проект × дата`. Заменяет «лид-в-канбане»
из текущей модели. Один канал может быть одновременно в N размещениях разных
проектов. Канбан проекта = канбан размещений.

**Артефакт** — file/URL, привязан к размещению, со своим статусом
(`получен → одобрен агентством → одобрен клиентом`). Типы: договор, драфт текста,
креатив, ЕРИД, опубликованный пост, скрин статистики 24ч/48ч, акт.

**Транзакция** — финансовая операция, привязана к размещению. Два направления:
- Клиент → агентство (предоплата/постоплата по договору)
- Агентство → блогер (по согласованной цене)
Маржа = разница, агрегируется в проект и в клиента.

**Conversation (Диалог)** — тред переписки `(blogger × project × manager × outreachAccount)`.
Один блогер одновременно имеет N открытых conversation'ов из разных проектов
разных менеджеров. История блогера = union всех его conversation'ов; в карточке
блогера видна общая timeline со всеми менеджерами и проектами в одной ленте.
TG-iframe в карточке блогера показывает switcher: «открыть тред от имени какого
проекта». Outreach-аккаунт назначается per-проект, не per-блогер: разные проекты
могут писать одному блогеру с разных TG-аккаунтов; для блогера это разные
«менеджеры», для агентства — единая история.

## Что из текущего кода переиспользуем

- **TG-iframe + сессии** (`apps/tg-client`, `outreach-account-client.ts`,
  two-session provisioning) — переписка с блогером из карточки канала.
- **Outreach worker + sequences** — становится подкапотным механизмом
  «pricing-request» из проекта. Sequence создаётся при запуске запроса прайса.
- **Inbound listener** — расширяется: парсит ответы блогеров (прайс) → апдейтит
  размещение в активном проекте.
- **Workspace + auth + UI каркас (sidebar/layout)** — без изменений.

## Что выкидываем или жёстко переделываем

- **Кастомные properties контакта как основной механизм** — для блогеров и
  каналов жёстко-структурированные поля (нужны индексированные фильтры).
- **Один глобальный канбан** → канбан-внутри-проекта.
- **Lists/CSV-импорт лидов** → остаётся как one-time pополнение базы блогеров,
  не как источник кампаний.
- **Outreach Sequences с активацией как отдельная фича** → подкапотный механизм
  внутри проекта, без отдельного UI-раздела.
- **`outreach_leads`** → заменяется на временную таблицу-черновик медиаплана.

## Что добавляем нового

- **Финансовый слой**: Транзакция, агрегаты маржи на уровне Размещение → Проект → Клиент.
- **Артефакты**: file storage + статусы согласования.
- **Парсер метрик каналов**: дёргает пост-URL через 24ч/48ч/7д, пишет в Размещение.
- **Генератор отчётов клиенту**: PDF/Google-Doc по шаблону по всем размещениям проекта.
- **Матчинг каналов по брифу**: рекомендательный слой поверх базы каналов
  (фильтры тематика/гео/бюджет).
- **ЕРИД-генератор**: на стороне агентства, в карточке размещения.

## Roadmap (плоский список фаз)

Каждая фаза — самодостаточный кусок работы. После фазы система остаётся в
рабочем состоянии (не разваливается).

### Фаза P0 — заморозка legacy + база

- Помечаем legacy-разделы (sequences, lists, contacts/$id как карточка лида)
  как «v1» в UI: серый бейдж «Legacy», ссылка на этот документ.
- Sidebar: добавляем секцию «Агентство (v2)» под legacy-секциями.
  Внутри пока пустые placeholder'ы.
- Всё что было — продолжает работать. Новые сущности просто не добавляются.

### Фаза P1 — Клиенты + Проекты (MVP)

- Schema: `clients`, `projects` (минимум полей: name, status, brief jsonb)
- API: CRUD ручки `/v1/.../clients`, `/v1/.../projects`
- UI: список клиентов, страница клиента, создание проекта, страница проекта
  (пока без медиаплана, просто визитка)
- Клиент = юр.лицо рекла, бриф = текст/файл, без структуризации полей пока

**Definition of done:** в воркспейсе можно создать клиента, под клиентом —
проект, у проекта виден бриф. Пайплайна ещё нет.

### Фаза P2 — База блогеров и каналов

- Schema: `bloggers`, `channels` (channels.bloggerId FK)
- API: CRUD + импорт CSV каналов (как outreach lists сейчас, но в реестр)
- UI: разделы «Блогеры» и «Каналы» в sidebar секции «Агентство», фильтры,
  страницы карточек
- Поля канала: тематика/гео/прайс/ER (структурированные select/number, не jsonb)

**Definition of done:** есть база каналов с фильтрами по бюджету/тематике,
видно к какому блогеру привязан канал, у блогера видны его каналы.

### Фаза P3 — Размещения + канбан проекта

- Schema: `placements` (channel_id, project_id, status, scheduled_at, properties jsonb)
- Стадии размещения как enum (см. список выше). Возможна правка перед запуском.
- Schema: `conversations` (blogger_id, project_id, manager_id, outreach_account_id,
  last_message_at) — фиксирует кто-с-кем-под-каким-аккаунтом общался по проекту.
- API: создание placements пачкой из реестра каналов в проект; lookup conversation'ов
  по blogger_id для card-timeline.
- UI: страница проекта получает канбан размещений, drag-and-drop стадий,
  клик по карточке = модал размещения; карточка блогера показывает общую timeline
  всех conversation'ов (любого менеджера/проекта), TG-iframe со switcher'ом
  «открыть тред от имени проекта X».

**Definition of done:** менеджер открывает проект, тянет каналы из реестра в
медиаплан, видит канбан, может двигать карточки по стадиям руками. Открыв карточку
блогера, видит полную историю переписки от других менеджеров/проектов.

### Фаза P4 — Финансы

- Schema: `transactions` (placement_id, direction `inbound|outbound`,
  amount, currency, status, paid_at)
- API: CRUD + агрегация по проекту/клиенту
- UI: финансовая вкладка в проекте (план/факт/маржа), сводка по клиенту

**Definition of done:** для каждого размещения видно цену блогеру и доход с
клиента; на проекте — итоговая маржа.

### Фаза P5 — Артефакты на размещении

- Schema: `placement_artifacts` (placement_id, type, url/file_ref, status)
- File storage: пока local fs / Yandex Object Storage stub
- UI: загрузка/просмотр в карточке размещения, статусы согласования

**Definition of done:** к размещению можно прикрепить договор, драфт, креатив,
скрины; видно что согласовано/что нет.

### Фаза P6 — Запрос прайсов из проекта (использует текущий outreach-worker)

- В UI проекта кнопка «Запросить прайсы»: выбор размещений в стадии «Подбор»
- Создаёт под капотом sequence с шаблоном «Привет, новый проект, твоя цена?»
- Inbound listener расширен: парсит ответы (price extraction LLM или regex),
  пишет в `placement.properties.price`, двигает стадию на «Прайс получен»,
  обновляет соответствующий `conversation.last_message_at`.
- Опционально (флаг проекта): не отправлять блогеру если у него уже открыт
  активный conversation в другом проекте — иначе блогер запутается между двумя
  параллельными переписками от разных «менеджеров».

**Definition of done:** менеджер кликает кнопку → блогерам уходят запросы
от outreach-аккаунтов → ответы автоматом попадают в карточки размещений.
В карточке блогера видно что в этот момент с ним идёт переписка по проекту X
(параллельно с историческими conversation'ами других проектов).

### Фаза P7 — ЕРИД + парсер метрик публикаций

- ЕРИД: интеграция с ОРД (или ручной ввод), генерация в карточке размещения
- Парсер метрик: cron-ручка раз в N часов, дёргает пост-URL через
  TG-клиент, считает охваты/ER, пишет факт-метрики в размещение
- UI: в размещении виден ЕРИД-токен (копи-паст), факт-метрики после публикации

**Definition of done:** агентство получает ЕРИД для размещения, после публикации
видит фактические охваты без ручного ввода.

### Фаза P8 — Постотчёт клиенту

- Шаблон отчёта (PDF или Google Doc): по всем размещениям проекта,
  агрегированные метрики, маржа
- Генератор: кнопка «Сформировать отчёт» на проекте

**Definition of done:** менеджер кликает → получает готовый отчёт по шаблону.

### Фаза P9 — Депрекация v1

- Удаление legacy-разделов (sequences, lists как самостоятельные разделы)
- Миграция: outreach_accounts остаётся как «TG-аккаунты» в общем разделе
  настроек воркспейса, не привязан к outreach-флоу
- Очистка кода: удаление старых таблиц после миграции данных или их дропа

**Definition of done:** в продукте только агентский флоу, legacy-код удалён.

## Что НЕ делаем сейчас (явно отложено)

- Multi-проект-в-один-канбан view (типа «все размещения по всем активным проектам в одной таблице»)
- AI-suggester медиаплана по брифу (даём фильтры, выбор ручной)
- Интеграция с биржами Telega.in / Lemma — внешние источники прайсов
- Multi-currency conversion на лету (фиксируем в той валюте, что согласована)
- Gantt-view медиаплана (только канбан в P3, gantt — позже если упрёмся)
- Multi-tenant white-label (агентство как клиент CRMChat)
