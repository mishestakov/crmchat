# specs/ — как пишем TЗ

Свод правил, чтобы все `US-*.md` выглядели одинаково и машинно проверялись.

## Источники правды

| Что | Где |
|---|---|
| Что пользователь делает | `user-stories.md` (product-facing) |
| Декомпозиция стори до кода | `specs/US-N.md` (этот каталог) |
| Схемы RPC (input/output, HTTP, used by) | `specs/api-contracts.md` (автосгенерировано) |
| Авторизация и прочие cross-cutting | `specs/<topic>.md` (`auth.md`, и т.д.) |
| Машинный инвентарь | `code-inventory.json`, `scope.json` |

Стори читают продакты. Контракты читают разработчики. Не смешиваем.

## Структура `US-N.md`

```
---
story: US-N
title: <короткий заголовок>
stage: N (название этапа)
coverage:
  routes: [RouteImport, …]              # из code-inventory.routes
  rpc:    [proc.name, …]                # только вызванные действием пользователя
  firestore: [fn, …]                    # функции из lib/db/*
  postmessage: [type, …]                # iframe-события
  ui_strings: [i18n.key, …]             # читаемые пользователем строки
---

# US-N · <заголовок>

**Предусловие**: (опционально, напр. "аутентифицирован — см. `auth.md`")

## Цель
Один абзац, зачем эта стори.

## Точки входа
Где и как пользователь запускает сюжет. URL и/или UI-путь (сайдбар → …).

## Форма (если есть)
Таблица полей: label, placeholder, валидация, обязательность. Короткими фразами.

## Что вызывается
Список RPC по именам. Сигнатуры — ссылкой на `api-contracts.md`, не инлайном.

## Success-flow
3-5 пунктов: что происходит после удачного действия (редирект, toast, обновление списка).

## Error-flow
Таблица "условие → что видит пользователь". Retry-политику тоже сюда.

## Скриншоты
Ссылки на `tools/capture/processed/routes/<id>/*.png`.

## Открытые вопросы
Факты, которых в capture нет и надо выяснить отдельно. Нумерованы: OQ-1, OQ-2…

## Критерии приёмки
Чек-лист на 4-6 пунктов, без нумерации. Бинарные проверки.
```

## Правила, выведенные кровью

### Границы стори
RPC-список включает **только вызовы, триггернутые действием пользователя внутри стори**. Если destination-страница после редиректа сама дёргает что-то на своём бутстрэпе — это **не часть этой стори**, это принадлежит той стори, чья это страница. Допустимо добавить одну строчку-примечание "destination дёргает X, см. US-M".

### Уровень детализации
TЗ — каркас, а не побитовая копия кода. Целевой объём одной стори — **~60 строк**. Если получается 150+ — режем:

- JSON-тела ответов → в `api-contracts.md`.
- Посхогу, sleep-ы, retry-логика внутри мутации → имплементация, не контракт.
- Низкоуровневые детали транспорта (tRPC batch, OPTIONS preflight, nanoid-длина) → в `api-contracts.md` или в README транспорта.

### Опечатки и баги в UI
Если нашли в проде опечатку — в TЗ пишем **как должно быть**. Отдельных bug-файлов не заводим: исправление — дело следующей итерации, TЗ описывает целевое состояние.

### Скриншоты
Берём из `tools/capture/processed/routes/<id>/` — там пары `*.png` + `*.html`. Обычно хватает трёх: entry (nav+500), settle (после всех запросов), modal/error (если применимо).

### Cross-cutting
Авторизация, глобальные layout'ы, sidebar, уведомления — **не дублируем в каждой стори**. Выносим в отдельный `specs/<topic>.md` и ссылаемся одной строкой в `Предусловие`.

### Открытые вопросы
OQ — это факты, которые мы не смогли зафиксировать в capture. Не "надо обсудить с продактом", а "нужен негативный capture" / "нужен ручной клик по этому пути". Если OQ превратился в "надо подумать" — он не здесь, он в бэклоге.

## Как валидируется полнота

`npm --prefix tools/capture run coverage` выполняет 5 проверок:

| | Что проверяет |
|---|---|
| A1 | Каждый keep-route упомянут в какой-то `US-*` (или помечен layout/deep-link в `ROUTE_EXEMPT`) |
| A2 | Каждый proc ∈ (stories ∪ `scope.rpc_decisions`) |
| A3 | Каждая firestore-fn ∈ `scope.firestore_decisions` |
| A4 | Каждый postMessage.type упомянут в `chat-spec.md` |
| A5 | Каждая i18n-строка из `ui-strings.json` либо в прозе, либо в `ui-ignore.json` |

Если появился новый proc/route/firestore-fn — либо добавляем в стори, либо в `scope.json` с `decision: exclude` и **осмысленным** `reason`. Не "не нужно".

## Порядок работы над новой стори

1. Открыть `user-stories.md` — прочитать US-N.
2. Открыть capture: `tools/capture/processed/routes/<RouteImport>/` — увидеть что реально дёргалось.
3. Сверить с `reconstructed/src/routes/...` — найти компонент, форму, мутации.
4. Заполнить шаблон выше. Объём — ~60 строк.
5. `npm run coverage` — не должен жаловаться на новые gaps в A1/A2.
6. Если появился новый proc без stories — решить decision в `scope.json` или вписать в эту (или смежную) стори.
