## 1. Копирование пригласительной ссылки не работает

**Тезис.** Кнопка «Скопировать» у инвайта молча ничего не делает — ни ссылки в буфере, ни ошибки, ни подтверждения. Заодно интерфейс инвайтов ощущается перегруженным — пересмотреть, всё ли там нужно.

**Как устроено.** Два места вызывают `navigator.clipboard.writeText()` без try/catch, без фолбэка и без визуального подтверждения:
- `apps/web/src/routes/_authenticated/w/$wsId/settings/workspace/invite.tsx:65`
- `.../settings/workspace/index.tsx:639` (список pending-инвайтов)

Clipboard API молча падает вне secure context (http://, не-localhost хост) и при отсутствии permission. Минимум: try/catch + фолбэк (выделяемое поле с ссылкой) + тост «Скопировано».

Интерфейс: форма (username + роль + «Создать ссылку»), после создания карточка с двумя кнопками («Копировать», «Открыть @username в Telegram»), плюс отдельный список pending-инвайтов со своими «Скопировать»/«Отказать». Кандидат на упрощение — возможно, одной плоской панели достаточно.

## 2. Username в инвайте: зачем он и сверяется ли при входе

**Тезис.** При создании инвайта обязательное поле username приглашаемого. Нужно ли оно? Если да — проверяется ли при акцепте, что авторизовались тем же TG-аккаунтом, и есть ли ошибка при несовпадении?

**Как устроено.** Сверки **нет, и это осознанное решение**, зафиксированное комментарием в схеме (`apps/api/src/db/schema.ts:146-148`): `telegramUsername` — только hint пригласившему «кому я отправляю»; при accept (`apps/api/src/routes/invites.ts:286-333`) любой залогиненный пользователь со ссылкой становится member'ом. Обоснование в комменте: жёсткая сверка ломается, если invitee сменит username.

**Открытое.** Решение «не сверять» осталось невидимым для пользователя: поле выглядит как защита, но ею не является. Варианты: (а) оставить как есть, но в UI инвайта подписать «для справки, не проверяется»; (б) сделать мягкую сверку — при несовпадении username показывать предупреждение «приглашение адресовано @X, вы вошли как @Y», с возможностью продолжить; (в) жёсткая сверка. Решить продуктово.

## 3. Страница инвайта не объясняет, что делать

**Тезис.** Попадая на страницу инвайта, человек не понимает, что нужно авторизоваться в Telegram (и под каким аккаунтом). Нужна явная инструкция и понятные ошибки.

**Из разговора.** Юля сразу запуталась (0:01:35–0:04:40): «мне нужно сразу с рабочего авторизовываться?», путала аккаунт для входа с аккаунтами для переписки, собиралась пересылать сообщение вместо открытия ссылки в браузере. Это два разных понятия — «аккаунт-логин» и «аккаунты-рассыльщики» — и интерфейс их никак не разводит.

**Как устроено.** `apps/web/src/routes/accept-invite.$wsId.$code.tsx`: незалогиненного редиректит на `/login?next=…` — то есть человек видит общую страницу логина без контекста «вас пригласили в воркспейс X». Карточка с «Принять/Отклонить» показывается уже после логина.

**Что сделать.** Пронести контекст инвайта на страницу логина («Вас пригласили в N. Войдите через Telegram — подойдёт любой ваш аккаунт, это аккаунт для входа, не для рассылок»). Поскольку сверки username нет (п. 2), сообщение об ошибке «не с того аккаунта» сейчас невозможно в принципе — связано с решением по п. 2.

## 4. Бот не различает регистрацию и вход

**Тезис.** После подтверждения в боте новому пользователю надо говорить «вы зарегистрировались, вернитесь в браузер», а существующему — «вы вошли». Сейчас текст один, флоу регистрации и авторизации не разграничены.

**Как устроено.** `apps/api/src/lib/tg-bot.ts:274-305`: на approve всегда один upsert (`onConflictDoUpdate`) и один текст «Вход подтверждён ✅ Вернитесь в браузер.» Различить просто: upsert вернёт/проверит существование юзера до вставки → две ветки текста. Также вопрос-кнопка «Подтвердить вход как …» для нового пользователя по смыслу — «Создать аккаунт и войти».

**Тезис.** Непонятно, на каком уровне должно жить расписание рассылки и видят ли его менеджеры. Рабочая гипотеза — на воркспейс (все по Москве).

Нужен сразу «богатый» интерфейс (как после старта или лучше): превью канала + поиск/назначение контакта. Проверить: ушла ли бы проблема «пришлось удалить каналы без админа», если бы он был.

**Из разговора.** 0:15:32–0:18:43: у 3 каналов из 30 LLM не нашёл админа, ткнуть в строку нельзя («ни по одному не даёт»), Юля ходила смотреть каналы руками в Telegram, находила контакты в описании («вопросы сюда», «фидбэк-бот»), но внести их в систему было некуда — каналы удалили. Миша по ходу: «это косяк, я UX правил… интерфейс на повороте удалил случайно».

## 7. След от удаления: пометка «Контакт не найден»

**Тезис.** Если канал удаляют из списка из-за ненайденного контакта, должен оставаться след, чтобы при повторном попадании канала в любой будущий список сразу было видно: «уже смотрели, контакта нет». Плюс нет массового удаления «всех без контакта» и фильтра по ним.

Канал как сущность (`channels`) при этом остаётся — след логично вешать туда (поле/свойство вида `contact_lookup_status: not_found` + дата), и показывать бейдж при следующем импорте. ## 8. Запуск рассылки: кнопка на другой странице, неактивна без объяснений → визард

**Тезис.** Кнопка запуска живёт не на странице списка, а в настройках, и когда она disabled — причина не показывается (Миша сам не понял, что не настроена цепочка). Переделать в пошаговый флоу с явной валидацией: 1) загрузил список → 2) у всех есть контакт (тут обязателен поиск/превью, см. п. 6) → 3) аккаунты → 4) цепочка сообщений → 5) запуск. Пока обязательный шаг не закрыт — дальше не пройдёшь, и видно, что именно не готово.

**Как устроено.** Кнопка — `apps/web/src/routes/_authenticated/w/$wsId/projects/$projectId/index.tsx:274-284`, `disabled={activate.isPending || messages.length === 0}`, причина никак не выводится (только opacity). При этом бэк на `/activate` валидирует больше и с внятными текстами (`apps/api/src/routes/projects.ts:536-608`): нет сообщений, нет активных аккаунтов, пустой список, «каналов без контакта — N». Т.е. фронт показывает меньше, чем бэк уже умеет сказать. Минимум до визарда: вывести чек-лист готовности (те же 4 проверки) прямо над кнопкой и продублировать кнопку запуска на странице списка.

## 9. Переменные шаблонизации: «канал, каналы, ссылка — что выбрать???»

**Тезис.** В пикере переменных три канальные переменные + username, и непонятно, какую брать. По ощущению нужна одна. Разобраться, есть ли путаница и лишний код.

**Из разговора.** Юля при составлении приветствия зависла именно на этом (0:20:00): «вот это юзернейм канал, название канала, нет, ссылка на канал, во, вот это надо» — выбирала методом тыка и потом переспрашивала «он точно подтянет ссылку?».

## 10. Цепочка сообщений: follow-up только после сохранения, дефолт «через 0 часов»

**Тезис.** (а) Кнопка «Добавить сообщение в цепочку» появляется только после первого сохранения — до этого непонятно, что цепочка вообще бывает. (б) Дефолт задержки follow-up — 0 часов, это опасно; ставить 3 часа. (в) Убедиться, что догоночное не уйдёт ночью.

**Как устроено.**
- (а) `apps/web/src/components/messages-editor.tsx:91` — «+ Добавить сообщение» рендерится при `value.length > 0`, т.е. только после сохранённого первого. Поправимо: показывать кнопку всегда, авто-сохраняя первый драфт, либо дизейблить с подсказкой.
- (б) Дефолт `{ period: "hours", value: 0 }` — `messages-editor.tsx:37`. Поменять на 3 часа — однострочник.
- (в) Ночью **не уйдёт**: follow-up планируется как `now() + delay` без учёта расписания (`apps/api/src/lib/outreach-worker.ts:556-576`), но worker отправляет только внутри окна воркспейса (`outreach-worker.ts:232`), так что «23:00 + 3 часа» проедет в 10:00 утра. Поведение правильное, фиксить нечего; стоит лишь отразить в UI («отправится в рабочие часы»).
- Бонус: follow-up'ы холодной цепочки корректно отменяются при ответе лида (`apps/api/src/lib/outreach-listener.ts:196-226`) — то, что Юля уточняла на 0:23:08, работает.

## 13. Очереди отправки: разобраться, как устроено, и решить, менять ли

**Тезис.** Сообщения уходят с двух аккаунтов параллельно — похоже на независимые очереди. Понять: как очереди устроены, что происходит при нескольких одновременных рассылках по разным проектам, и не будет ли при 50 аккаунтах подозрительной синхронной отправки «секунда в секунду». Пока ничего не менять — сначала понимание.

**Как устроено** (`apps/api/src/lib/outreach-worker.ts`, `apps/api/src/lib/project-scheduling.ts`):
- Очередь — таблица `scheduled_messages` (строка = item × message_idx, статусы pending/sent/failed/cancelled, индекс по `(status, sendAt)`).
- Worker: один глобальный тик каждые 10 с → выборка pending с `sendAt <= now` (до 500) → группировка по аккаунту → **максимум 1 сообщение на аккаунт за тик** → аккаунты обрабатываются параллельно (`Promise.all`).
- Human-flow на каждом сообщении: openChat → typing 15–40 с (рандом) → send → пауза 60–150 с (рандом) → closeChat. За счёт рандома фактические моменты отправки разных аккаунтов рассинхронизированы, хотя тик общий.
- Несколько кампаний: на уровне worker'а разделения нет — все pending данного аккаунта в одной куче, порядок по `sendAt`. Распределение каналов по аккаунтам происходит один раз при активации: sticky (если peer уже общался с аккаунтом / `contacts.primary_account_id`) → иначе round-robin (`project-scheduling.ts:140-185`).
- Лимиты: `outreachAccounts.newLeadsDailyLimit` (дефолт 15) — считаются только «холодные» первые сообщения за день в таймзоне воркспейса; тёплые без лимита. `hasPremium` — просто информационный флаг, **никакого ×2 в коде нет**.

**Оценка для 50 аккаунтов.** «Секунда в секунду» не случится из-за рандомных typing-пауз, но есть масштабные вопросы на потом: один процесс-worker на все аккаунты, общий батч в 500, отсутствие per-account джиттера старта тика. Менять сейчас нечего; вернуться при росте парка.

## 14. «Без стадии» на канбане: начальная стадия

**Тезис.** После первого контакта карточка падает в «Без стадии». Вроде была фича «отметить изначальную стадию» — поискать в коде и восстановить.

**Как устроено.** Фича в коде **не найдена** (поиск `initialStage`/`defaultStage` по всей git-истории пуст). `stageId` при создании item'а — `null`, автоматических переводов по событиям нет вообще: ни «отправили первое сообщение → Новый», ни «ответил → Ответил» (`apps/api/src/lib/outreach-listener.ts:196-207` ставит только `repliedAt`). Единственный способ сменить стадию — ручной drag-drop (`apps/api/src/routes/projects.ts:1072-1104`). Ближайшее существующее — дефолтный набор стадий проекта (`DEFAULT_OUTREACH_STAGES`, `schema.ts:599-604`), но это другое.

**Что сделать.** Видимо, воспоминание о «штуке в коде» неточное — проектировать заново: настройка проекта «начальная стадия» (куда падает карточка при отправке первого сообщения) + опционально авто-перевод в стадию при ответе. Минимум: при активации проставлять всем item'ам первую стадию проекта.
