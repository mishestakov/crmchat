# Обогащение «Каналы Яндекса» (CPC/CPA-активность)

Статус: согласовано продуктово, к реализации. Аналог РКН-обогащения, но
источник — рекламные платформы Яндекса (YT-кластер), а не Госуслуги.

## Зачем

У Яндекса есть свои рекламные платформы, где блогеры уже монетизируются:
**CPC** (`tgads`) и **CPA** (`cpa_network`). Подтягиваем знание «канал уже
крутится у Яндекса» и накладываем на наши списки каналов — как реестр РКН.

Два бизнес-сценария над одним датасетом → **один раздел в меню**, не два.
CPC/CPA — внутренняя кухня Яндекса; продукту важно «работает / не работает +
насколько свежо + в каком состоянии».

1. **Видеть, кто уже работает** (информ, делаем сейчас). При наборе аутрича на
   лиде показывается **бейдж** «работает на платформе» (CPC/CPA) — менеджер
   видит статус и решает сам. Это НЕ авто-гейт: отправку не блокируем.
   Почему не гейт: CPC/CPA-сигнал ненадёжен — за 2 года админ мог смениться, у
   одного админа часть каналов активна, часть нет. Источник —
   `platform-active.ts` (`channelAlreadyWorkingSql`), матч симметричный (см.
   ниже). Свежесть (активен/спал N мес) — содержимое бейджа, не фильтр.
2. **Win-back отвалившихся** (проактивный, потом). Канал был активен, но
   перестал публиковаться / бот потерял права — повод вернуть. Это источник
   лидов. Данные под него уже копим (даты + состояние + качество), саму
   поверхность набора делаем отдельным проходом. Гейта нет → спящие каналы
   доступны для win-back-аутрича (не выпадают, как было бы при авто-гейте).

## Как работает, end-to-end

1. **Суточный python-job** в YT: две YQL-выгрузки (CPC + CPA), нормализация,
   **bulk-replace** таблицы-зеркала с предохранителем (если выгрузка оборвалась
   / усохла > 2% — не применяем, оставляем прошлую). Штамп `last_sync_at`.
2. У каждого канала-из-Яндекса считается **массив отпечатков** (`@username` +
   id канала + инвайт-хэш). У наших каналов — такой же массив
   (`channel-match-keys.ts`). Матч **симметричный** (`pac.match_key &&
   candidates`): канал, известный пока только по @username (id появляется лишь
   после открытия карточки), всё равно находится.
3. **Бейдж**: наш канал, пересёкшийся по отпечаткам → бейдж «работает на
   платформе» на лиде. Аутрич НЕ блокируется (см. сценарий 1).
4. **Страница «Каналы Яндекса»** — браузабельный справочник по всему датасету
   (зеркало страницы РКН): поиск, фильтры, дата обновления. Точка входа для
   ручной проверки и будущего набора win-back.

## Архитектура синка

- **python-сервис в том же prod compose-network**, ходит в `postgres:5432`
  напрямую (как node) — Postgres наружу не торчит, доступ к YT с прод-VM есть.
- **`match_key` — generated-колонка `text[]`** (массив отпечатков), собирается
  тем же `channelMatchCandidatesSqlText`, что и сторона channels (один билдер,
  ноль дрейфа). python пишет только сырую идентичность — БД выводит массив на
  запись. Матч — пересечение массивов (`&&`), индекс GIN. Эндпоинт не нужен.
- node владеет **DDL + чтением** (страница + бейдж); python — запись + guard +
  штамп меты. Для python — **отдельный DB-юзер** с правом писать только в
  `platform_active_channels` + `platform_active_sync`.
- Расписание — **внешний планировщик** (systemd-timer / cron на прод-VM, где
  YT-токен), не in-process таймер как у РКН: данные и токен живут в
  python-окружении. Lock-файл + retry + алерт при падении.

## Данные (таблица `platform_active_channels`)

PK — `source_key` = `"<source>:<natural_id>"` (дедуп внутри выгрузки; одна
строка на запись-источник, без мержа). Страница показывает по строке на
источник: source-specific статусы (`bot_status` у CPC, `moderation_status` у
CPA) видны как есть, канал-в-обоих даёт две информативные строки. Мерж «оба» в
одну строку — косметика на потом.

| Группа | Поле | Источник | Назначение |
|---|---|---|---|
| Идентичность | `platform` | оба | telegram/youtube/tiktok/dzen/max (vk дропаем) |
| | `external_id` | оба | id канала в соцсети (TG `-100…`) |
| | `username` | оба | `@handle` без `@` |
| | `link` | оба | ссылка (вкл. инвайт `t.me/+…`, `max.ru/join/…`) |
| | `owner_login` | CPA | яндекс-логин владельца (для win-back) |
| | `match_key` | derived | generated `text[]`: массив всех отпечатков (GIN) |
| Система | `source` | — | `cpc` / `cpa` (на строку) |
| Активность | `last_post_date` | оба | детектор «отвалился» |
| | `recent_posts_count` | оба | постов за окно (60д) |
| | `recent_views` | оба | просмотров за окно (60д) |
| Состояние | `bot_status` | CPC | здоровье бота (`OK` / `ADMIN_RULES_TIMEOUT`…) |
| | `is_active` | CPC | активен в tgads |
| | `is_cpv` | CPC | **маркер качества** — допущен в режим CPV |
| | `moderation_status` | CPA | этап ЖЦ страницы («Закончился тест. период»…) |
| | `updated_at` | — | время записи |

Мета-таблица `platform_active_sync` (singleton): `last_sync_at`, `last_status`,
`total` — зеркало `rkn_sync`, штампует python при коммите.

`match_key` (generated `text[]`, STORED): `ARRAY[username-ключ, external_id-
ключ, tg-инвайт, max-инвайт]` — тот же `channelMatchCandidatesSqlText`, что у
channels. Матч симметричный (`pac.match_key && candidates`): хранятся ВСЕ
отпечатки, поэтому ни username-only, ни id-only канал не теряется. NULL-
элементы в `&&` безвредны.

## Страница «Каналы Яндекса»

Глобальная (не workspace), ручка `/v1/platform-active`, живёт в ws-шелле ради
сайдбара — зеркало страницы РКН.

Колонки:

| Платформа | Система | Канал | Статус | CPV | Последний пост | Постов 60д | Просмотры 60д |

- **Система** = CPC / CPA (на строку, не платформа!).
- **Статус** = бейдж: для CPC `bot_status`/`is_active`, для CPA
  `moderation_status`.
- **CPV** = бейдж «✓» если допущен (знак качества).
- Фильтры-чипы: платформа, система, статус, «только CPV»; поиск по
  `@username` / ссылке / логину владельца.
- Шапка: «Обновлено N назад · X записей · автообновление раз в сутки» + плашка
  при упавшем синке.

## Чего НЕ делаем (MVP-минимум)

Дропаем из выгрузки: `title`/название (у сматченных своё), внутренние YT-id
(`page_id`/`channel_id`/`page_key`), `earned_rub`, sandbox-флаг, **vk**/rutube/
прочие не-наши площадки. Без chunked-upload (полный снапшот за раз), без live-
прогресса синка на странице (только дата+статус), без мержа «both» в БД,
win-back-поверхность — отдельным проходом.

**CPA-нюанс:** `media_type` у CPA ненадёжен (под одним media_type лежат ссылки
на разные площадки), поэтому платформу и идентичность определяем **по хосту
URL** (`_platform_from_url`/`_cpa_identity`): t.me→telegram, youtube.com→youtube
(`/channel/UC…`→external_id, `/@handle`→username), dzen.ru/tiktok.com→handle,
max.ru→инвайт; vk/ok/rutube → дроп. CPC — по надёжному полю `Platform`.
