## 1. Дубли при подтверждении + таблица по контактам

**Проблема** (0:23:57–0:26:24). Блогер/админ с несколькими одобренными каналами получает 2–3 одинаковых «вы выбраны». «Отправить 3 блогерам» — по факту 2 получателя. Надо: (а) дедуп — одно сообщение на контакт, (б) UI-таблица **по контакту**, каналы через запятую.

**Как сейчас**
- Модель: 1 `project_item` = 1 канал. Два канала одного админа → два item'а с **одинаковыми** `contactId/username/tgUserId` (`campaigns.ts:668-683` `resolveAdminRecipient`).
- POST `/final-offer` (`campaigns.ts:861-999`) в цикле `rows.map` (`:945-965`) кладёт **по одному `scheduled_message` на item** — без дедупа по получателю. Воркер (`outreach-worker.ts:336`) шлёт каждый на `tgUserId` item'а → дубли.
- Guard `NOT EXISTS` от повторной рассылки — тоже по `itemId` (`:914-919`), не по контакту.
- Счётчик на фронте по каналам: `notOffered` (`$campaignId.tsx:1675-1679`) → лейбл «Отправить N блогерам». Таблица — строка на канал (`:1750-1795`).

**Что менять**
- **Backend (`campaigns.ts` handler final-offer, `:945-965`):** перед `rows.map` схлопнуть `rows` по ключу получателя (`tgUserId ?? username ?? contactId`), оставить одного представителя → один `scheduled_message` на админа. Ответ `{ scheduled }` станет = числу уникальных получателей.
- **Guard (`:914-919`):** расширить до «есть sent/pending оффер у ЛЮБОГО item'а этого контакта», иначе повтор нажатия снова пошлёт админу, у которого оффер ушёл через другой канал.
- **Frontend (`ShortlistPhase`, `$campaignId.tsx:1675-1795`):** группировать `approved` по `adminContactId` (fallback `id`); строка = `Контакт | Каналы (через запятую) | Подтверждение (статус представителя)`. `notOffered` считать по уникальным контактам. Данные уже в API (`adminContactId`, `adminUsername`, `channel.*`, `hasRecipient`, `finalOfferStatus`) — контракт не меняем.

**Риски**
- `finalOfferStatus` derive per-item (`:337`): у «братских» каналов без своего `scheduled_message` останется `none` → перегруппировка UI по контакту не косметика, а необходимость (статус берём с представителя). Грепнуть `finalOfferStatus` по web — не читается ли где-то ещё per-item (фаза «Запуск»).
- Текст оффера `substituteVariables` (`:958-962`): подтвердить, что нет канал-специфичных плейсхолдеров (при дедупе берём представителя).

## 2. Пометка креатива «не зеленеет / не обновляется»

**Проблема** (0:26:24–0:27:13). Пометил сообщение как «Креатив» — шаг не позеленел, рефреш не помог. Продукт: разрешить помечать креатив **до** подписания договора.

**Как сейчас — разбор (важно: это на 80% by-design + 1 реальный баг раскладки)**
- Пометка (`tagMut` → PUT `/step-message/creative`) двигает `creativeStatus` `none/awaiting → internal_review` (`campaigns.ts:1472-1474`). В `approved` не переводит.
- Шаг «Креатив» зеленеет только при `creativeStatus === "approved"` (`-placement-drawer.tsx:1286,1400`), а `approved` ставит **только клиент** в share-портале (`share-client.ts:629`). Т.е. «зелёный» после пометки — **ожидаемо не появляется**.
- Инвалидация корректная (`tagMut.onSuccess` бьёт `["placements",…]` + `["step-message",…]`, `:1049-1054`) — «нужен рефреш» этим **не** объясняется.
- **Реальный баг:** степпер секвенциальный. `currentIdx = doneServer.findIndex(!done)` (`:1402`), `doneServer[0]` = «договор подписан И акт получен». Договор пропущен → `currentIdx=0`, шаг «Креатив» = `future` → **свёрнут**, тело (превью + «Показать клиенту») скрыто. После пометки визуально ничего, рефреш не помогает (текущий шаг всё ещё Документы).
- Бейдж пометки в чате — **фиолетовый** (`chat-drawer.tsx:1228`), не связан цветом с галочкой степпера → усиливает ощущение «не сработало».

**Что менять**
- **Разорвать зависимость креатива от договора в степпере** (`-placement-drawer.tsx:1402-1403, 1432-1490`): раскрывать/показывать шаг «Креатив», как только есть `stepMessages.creative`, независимо от `done` предыдущего. Сама пометка ничем не гейтится — правка чисто во фронтовой раскладке.
- **Видимый промежуточный статус** на кружке/подписи для `internal_review`/`client_review` (сейчас серый до `approved`) — чтобы «пометил» читалось как прогресс. Согласовать цвет с фиолетовым бейджем чата.
- **Живой рефреш чинить не надо** — уже работает.

**Продуктовый вопрос:** считать шаг «Креатив» завершённым при `internal_review` (менеджер пометил) или только при клиентском `approved`? От этого — цвет кружка и «этап Запуск завершён».

## 3. Картинки постов не грузятся в клиентском вью

**Проблема** (0:21:09). В клиентском (magic-link, без auth) вью «последние посты» — картинки не грузятся. Идея: локальный кэш на сутки; про TDLib `use_file_database`.

**Как сейчас**
- Клиентский дровер (`share.$token.tsx:387-401` → `channel-preview-drawer.tsx`) вызывается **без** `wsId/channelId` → `Post` (`channel-card.tsx:1641`) не строит full-res URL и рендерит **только блюр-минитамбнейл** (`MessageMediaThumb`, base64 ~250B). Full-res клиент не видит **никогда**.
- Full-res роут `/channels/:id/post-media/:messageId` (`channels.ts:1946`) — под `assertMember` + `pickChannelReader`, клиенту недоступен. Share-scoped есть только для **креативов** (`share-client.ts:573-600`), для постов канала — нет.
- Премиса «клиент не может тянуть медиа» верна только для браузера: **сервер может** через outreach-аккаунт воркспейса (как уже делают `preview` и `creative-media`). Свой TDLib клиенту не нужен.
- `use_file_database` — **уже включён неявно** через `use_message_database:true` (`client.ts:77-86`); влияет только на персистентность метаданных серверного TDLib между рестартами. **Баг не решает** (не даёт неавторизованному браузеру байты). Честный вывод: ортогонально.
- Кэша full-res медиа постов нет (`channel_thumbnails` кэширует только аватар канала).

**Что менять (правильный путь — server-side кэш, cache-only для клиента)**
1. Новый токен-роут `share-client.ts`: `GET /v1/share/{token}/placements/{placementId}/post-media/{messageId}` (scope как в `preview`, `:401-416`).
2. Параметризовать построение media-URL в `Post`/`ChannelPreviewDrawer` (сейчас хардкод `/v1/workspaces/...`, `channel-card.tsx:1669`), чтобы клиентский дровер включал ветку `fullRes` с share-URL.
3. Кэш: файлы на volume (`TDLIB_DATA_DIR`) + индекс-таблица `channel_post_media(channelId, messageId, path, updatedAt)`, TTL-свип 24ч.
4. **Attack-surface:** наполнять кэш **со стороны менеджера** (когда открывает ленту — `/post-media` уже качает байты, там же класть в кэш). Клиентский эндпоинт отдаёт **только из кэша**: miss → 404/блюр, **никогда** не дёргает `downloadFile`. Тогда публичный токен физически не гоняет TDLib.

**Риски**
- Уточнить на стенде, что видит клиент: блюр-заглушки (вывод по коду) vs пустой дровер (если у ws нет активного telegram outreach-аккаунта — `preview` вернёт `[]`).
- Приватные каналы без подписки outreach-аккаунта — full-res недоступен в принципе.
- Диск: TTL-свип + лимит на канал, иначе volume забьётся. Видео отдаётся постер-кадром (как `creative-media`).

## 4. ЕРИД: данные рекламодателя не подставляются + хранятся на 2 уровнях

**Проблема** (0:07:05, 0:34:45). ИНН/название рекламодателя должны тянуться из настроек кампании в ЕРИД-маркировку. Плюс данные задаются **и на уровне клиента** — зачем 2 уровня? Желаемый формат:
```
Маркировку наносим на видео/фото небольшим, но заметным шрифтом:

Реклама. Рекламодатель ООО «Инстамарт Сервис», Москва, ОГРН 1187746494980, ERID: 2VtzquYTh8n
```

**Как сейчас**
- **2 несвязанных хранилища** (продукт-оунер прав):
  - Кампания — `projects.advertiserData` (свободный текст «ИНН + название», `schema.ts:719-721`) — **единственный**, что реально идёт в ЕРИД.
  - Клиент — `tracks.properties` jsonb (`legal_entity/inn/contract/…`, `schema.ts:571-593`, форма `client.$clientId.tsx:16-23`) — **никак не связан** с ЕРИД, отдельная справочная карточка. Менеджер вводит ИНН клиента дважды.
  - (+ per-placement override `projectItems.eridAdvertiserData`.)
- **Подстановка на самом деле УЖЕ работает:** `eridAdv = draft.eridAdvertiserData || advertiserData || ""` (`-placement-drawer.tsx:975`) и шлётся в чат (`:1072`). Но инпут ЕРИД-шага держит `value=""`, а бриф-значение только как **placeholder** → поле выглядит пустым → ощущение «не подставляет».
- **Формат не тот:** текущий шаблон (`:1072`) — `ERID: … / Рекламодатель: … / Нанесите…`. В желаемом — `ОГРН` и город, а таких полей в схеме **нет** (есть только ИНН-текст).

**Что менять**
1. **Single source — уровень клиента** (`tracks.properties`; клиент постоянен между кампаниями). Наследовать в кампанию/ЕРИД, убрать ручной дубль `projects.advertiserData`. Добавить недостающие `ogrn`, `city` в `tracks.properties` (jsonb — миграция не нужна).
2. **Шаблон маркировки** — переписать `-placement-drawer.tsx:1072` под точный образец, вынести формат в helper (`@repo/core`/`-shared.ts`) как single source.
3. **Синхронизировать инпут:** инициализировать `eridAdvertiserData` дефолтом (видимое значение, не только placeholder).

**Продуктовые вопросы**
- В образце **ОГРН, не ИНН** — оставлять ли ИНН в маркировке? Добавляем поля `ogrn`, `city`?
- Убрать `advertiserData` с кампании совсем или оставить как override? (per-placement `eridAdvertiserData` — сохранить как исключение.)
- Два ws-режима (`bd`/`agency`) хранят реквизиты клиента по-разному — унифицировать оба.

## 5. Метрики канала не тянутся до открытия превью-дровера

**Проблема** (0:10:07, 0:18:00, 0:39:09). Ср.охват/ERR/прогноз/подписчики пусты, пока менеджер вручную не прокликает канал (глаз «канал»). Надо: тянуть при **открытии строки** лонглиста.

**Как сейчас (точная цепочка)**
- `avg_reach`/`err` пишет **только** `GET /history` (`channels.ts:1816-1835`, гейт `deepMetrics`, TTL 24ч по `meta.metrics_at`).
- `/history` зовёт **только** `PostsFeed.initialQ` (`channel-card.tsx:1397-1415`), а `PostsFeed` живёт **только** внутри `ChannelCard` → `ChannelFeedDrawer` → открывается по клику (`-placement-drawer.tsx:305-339, 630`).
- `/sync` (автосинк `ChannelCard`) метрики **не** считает.
- `PlacementPane` при выборе строки делает только `channelQ = GET /channels/{id}` и читает `meta` как есть (`-placement-drawer.tsx:129-147`) — если скана не было, пусто.

**Что менять**
- В `PlacementPane` (рядом с `channelQ`, `-placement-drawer.tsx:129`) добавить headless `useQuery` с **тем же** ключом и `queryFn`, что у `PostsFeed.initialQ` (`["channel-history", wsId, channelId]`, `GET /history`, `limit:50`). Одинаковый queryKey → при открытии дровера второго TDLib-захода не будет (кэш). На `isSuccess` инвалидировать `["channel", wsId, channelId]` (как `channel-card.tsx:1443-1452`).
- **Обязательно** повторить `enabled`-гейты `PostsFeed` (`channel-card.tsx:1388-1393`): есть `externalId`; не в `unavailable`-cooldown; есть активный outreach-аккаунт платформы; **не** provider-платформа (для YouTube/TikTok/Dzen `/history` неприменим — там метрики пишет `/sync`).
- Опционально консервативнее: `enabled: … && avgReach === null` — сеть только для непросканированных.

**Риски**
- Флуд: авто-скан на добавлении убрали намеренно (`campaigns.ts:698-699`). Сдвиг триггера «открыл превью» → «выбрал строку» эагернее, но лениво (по одному каналу на явный выбор + TTL 24ч). Приемлемо.
- Provider-платформы: если для них тоже нужен авто-охват — звать `/sync`, не `/history` (подтвердить, что `syncChannelFromProvider` пишет `avg_reach`/`err`).
- UX-мерцание: показать плейсхолдер/спиннер в строке метрик на `isFetching`.

## 6. Дедлок: нет превью канала на этапе выбора админа (взять BD-UX)

**Проблема** (0:08:41). Чтобы назначить контакт админа, надо судить о канале — но превью на этом этапе недоступно. В BD-режиме сделано хорошо: превью + кандидаты в админы рядом. Хотим: нет админа → BD-вид (превью + кандидаты); указали → текущий вид (цены, чат).

**Как сейчас**
- `ContactResolver` (`contact-resolver.tsx:14`) канал использует только чтобы распарсить @-хендлы из описания и личку — **ни метрик, ни ленты, ни `channel.admins`**. Судить об админе не по чему.
- В `PlacementPane` при отсутствии способа связи резолвер занимает всю ширину (`-placement-drawer.tsx:609-628`), а кнопка превью рендерится **только** в deal-ветке (`hasMethod`, `:304-340`) → в no-method её нет. Отсюда дедлок.
- **BD-режим (`LeadPrepPane`, `lead-prep-pane.tsx:153-211`) делает ровно нужное:** слева `<ChannelCard compact>` (hero-метрики + лента постов), справа `ContactResolver` (кандидаты). Когда контакт готов — правая колонка морфится в «Контакт для рассылки». Тот же паттерн.

**Что менять (переиспользуем существующее, новых компонентов не плодим)**
1. `-placement-drawer.tsx:609-628` — заменить полноширинный `ContactResolver` на две колонки как `LeadPrepPane:153-211`: слева `<ChannelCard wsId channel={channelQ.data} compact />` (данные уже есть, тот же queryKey — дедуп), справа фикс-колонка с текущим `ContactResolver` (пропсы не меняются).
2. Deal-ветку (`hasMethod && !changing`, `:288-608`) **не трогаем**.
3. Опц. усилить «подмену админа»: в compact `channel.admins`/`suggested_admin` скрыты — если продукт хочет выбор из известных админов, прокинуть их чипами в `ContactResolver` (клик → `set-admin`). Общий рычаг у всех путей — `POST /channels/{id}/set-admin` (консистентно).

**Риски / вопросы**
- Ширина панели кампании — влезут ли превью (`flex-1`) + резолвер (`w-[360px]`).
- Что считать «кандидатами в админы»: `channel.admins` / `suggested_admin` / хендлы из описания (сейчас в BD-резолвере видны только последние + группа/личка/поиск).
- `suggested_admin` ставится только для каналов с активным размещением (`channels.ts:2920`) — в свежем лонглисте может не быть; проверить, что кандидат из импорта (`channel_admins`) доступен.
- Provider-каналы: в compact контактный блок Дзена скрыт — для них превью беднее.
